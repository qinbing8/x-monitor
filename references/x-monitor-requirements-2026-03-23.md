# x-monitor 需求与验收约束清单（2026-03-23）

## 0. 产品 invariant（不可偏离）

这是一个**指定 X 信息流筛查 skill**，不是人物资料补全 skill。

唯一正确目标：

**本地 CSV 名单 -> Grok 抓取名单对应博主过去 24 小时的 X 推文流 -> GPT 进行高价值信息筛查并产出日报。**

如果实现结果偏成以下任一方向，视为不合格：
- 人物 profile enrichment / 人物画像补全
- 关注者资料整理 / roster audit
- 账号标签分类而非推文流筛查
- 任何不以“过去 24 小时推文内容”为核心证据的方案

---

## 1. 角色边界

### Grok 只负责：抓取 + 整理原始内容
- 输入：本地 CSV 名单
- 工作：抓取名单里这些账号**过去 24 小时**在 X 上的推文内容
- 输出：干净、可审计、可复制的原始推文列表 / 结构化结果
- 禁止：
  - 做价值判断
  - 做主观分析
  - 做人物画像补全
  - 做“谁值得关注”的排序结论

### GPT 只负责：分析 + 筛选 + 日报写作
- 输入：Grok 抓回来的推文内容
- 工作：人工主编式筛查、去伪存真、给出日报
- 禁止：
  - 伪造 Grok 没抓到的内容
  - 把分析建立在 profile metadata 上而非推文内容上

---

## 2. 抓取范围（用户最终拍板）

时间范围：**过去 24 小时**

抓取内容包含：
- 原创推文
- 转发
- 引用推文

排除：
- 纯回复

注意：这是最终规则。不要再改回“只抓原创”，也不要改成“人物信息补全”。

---

## 3. Grok 输出要求（源规范）

Grok 提示词应保持与下述语义一致，可做最小必要工程化改写，但**不得改变任务目标**：

```text
你可以访问X/Twitter的实时数据。请执行以下任务：
## 任务
获取指定用户列表在过去24小时内的所有发帖记录，并以CSV格式输出。
## 用户列表

## 输出要求
CSV格式，包含以下字段：
username：发帖用户名
tweet_id：推文ID
created_at：发布时间（UTC）
text：推文完整内容（包含转发、引用内容）
original_url：推文链接
## 筛选条件
时间范围：过去24小时包含原创推文、转发、引用推文
排除纯回复
## 输出格式
直接输出可复制的CSV文本，首行为表头，使用英文逗号分隔，文本字段用双引号包裹。
请开始执行。
```

### 最低可接受字段
Grok 最终产物至少要能稳定提供：
- `username`
- `tweet_id`
- `created_at`
- `text`
- `original_url`

### 工程要求
允许在 fetch 阶段同时保留：
- 原始 CSV 文本（如 `fetch.raw.csv`）
- 归一化 JSON（如 `fetch.result.json`）

但**原始语义必须仍然是“24h 推文流”**，不是 profile records。

---

## 4. GPT 分析要求（源规范）

GPT 端应恢复/保持为“资深科技主编做推文筛选日报”的提示词语义。以下内容是需求锚点：

### 核心目标
- 逐条阅读推文
- 做 0-3 星质量判断
- 提炼高价值信息
- 输出日报

### 优先关注的高价值类型（用户优先级）
1. 深度观点 / 洞察
2. 新工具 / 新产品
3. 重大行业动态

同时可以保留：
- 优质资源

### 必须保留的输出结构
1. **今日要点摘要（Deep Brief）**
2. **编辑精选（Editor's Choice）**
3. **高价值推文完整清单**

### 必须保留的筛选逻辑
高价值信号：
- GitHub 仓库、Demo、Paper 解读
- 解决具体 Bug / 技术难题的经验分享
- 有数据支撑的行业分析
- 具体教程、Cheat Sheet、学习路径

噪音：
- 闲聊 / 情绪 / 吐槽
- 无意义回复
- 纯营销 / 空投 / 币圈推广（除非底层技术讨论）
- 看不懂上下文的碎片语句

---

## 5. 日报与调度要求

目标运行方式：**每天 8:00，Asia/Shanghai** 自动跑一次。

最终用户体验：
1. Grok 抓取过去 24 小时推文流
2. GPT 生成日报
3. 系统**主动把日报发给用户**

### 失败降级策略（用户最终拍板）
如果抓取失败或只覆盖部分账号：
- **仍然发日报**
- 但必须明确标注：
  - 哪些账号未覆盖
  - 哪些抓取失败
  - 哪些结果不完整

原则：**宁缺毋滥，不要猜。**

---

## 6. 低信噪比账号提醒

只有当某个账号**明显刷屏低价值内容**时，才在日报里提示。

不要把每个低质量账号都列出来；只在下列情况提示：
- 当日明显高频刷屏
- 且大多数内容被判定为低价值 / 噪音
- 已经影响整体信噪比

目标是帮助用户决定是否把该账号移出 Grok 抓取名单。

## 7. 对当前错误方向的处理要求

当前 repo 里已有一版错误方向的改造：把 skill 改成了 **CSV -> Grok 人物资料补全 -> GPT 人物画像分析**。

这次修复时：
- 可以复用其中**通用工程骨架**（例如 CSV 输入、批处理、artifact 风格、测试框架）
- 但必须移除或回退所有“profile enrichment”语义，包括但不限于：
  - SKILL 描述里的人物补全定位
  - fetch 阶段的人物资料 schema
  - analyze 阶段的人物画像 prompt
  - tests 中以 `profileId` / profile summary 为中心的断言

一句话：**保留骨架，不保留错误产品语义。**

## 8. 验收约束（必须全部满足）

### A. 语义验收
必须满足：
- skill 的核心对象是“推文流”，不是“人物资料”
- Grok 的 fetch 输出围绕 tweet items
- GPT 的 analyze 输入围绕 tweets / evidence block
- 最终日报围绕“今天发生了什么值得看”，不是“这些人是谁”

### B. 提示词验收
必须满足：
- `assets/prompts/grok-fetch.txt` 的任务中心是：**24h tweet fetching**
- `assets/prompts/gpt-analyze.txt` 的任务中心是：**主编式筛选日报**
- 不得再把 prompt 改造成 profile enrichment

### C. 配置验收
必须满足：
- `config.json` 中 fetch 侧配置应表达：CSV seed + 24h tweet fetch
- 如需 `batchSize`，可以保留
- 不得要求修改 skill 内部配置去支持人物资料补全

### D. 产物验收
至少应能稳定产出：
- fetch 阶段原始推文产物（CSV 或等价原始文本）
- fetch 阶段归一化推文 JSON
- analyze.input.json
- analyze.result.json
- final.md

### E. 测试验收
测试至少覆盖：
- CSV 种子读取
- Grok tweet CSV 解析 / 归一化
- GPT analyze 输入围绕 tweet evidence 而非 profile evidence
- 主流程 artifact 生成

### F. 边界验收
不得在本任务中做以下事：
- 不要修改 OpenClaw 核心配置（如 `~/.openclaw/openclaw.json`）
- 不要写入真实密钥
- 不要把“定时任务注册”伪装成已完成，如果它依赖核心配置或 cron 注册，则应明确分离为后续步骤

## 10. 最终自检（交付前逐条回答）

在交付前，必须先自问并确认全部为 YES：

- 这还是不是一个“24h 推文流筛查 skill”？
- Grok 抓的是不是推文，而不是人物资料？
- GPT 分析的是不是推文价值，而不是人物画像？
- 日报是不是在回答“今天有哪些值得关注的信息”？
- 如果用户只看 `final.md`，会不会感觉这是“日报”而不是“人物名单报告”？

只要其中任意一条不是 YES，就说明任务仍未对齐，需要继续修正。

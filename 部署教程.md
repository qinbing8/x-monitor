# x-monitor 部署教程

这份教程对应当前仓库的实际实现，目标只有一条：

- 电脑关机后，依然能每天自动生成并查看日报

当前采用的部署架构：

- `GitHub Actions`：定时执行抓取和分析
- `Cloudflare R2`：保存状态文件和日报产物
- `Cloudflare Workers`：用你的自定义域名展示日报

当前不使用：

- Render
- 数据库
- Telegram Bot

## 0. 先说清楚：哪些步骤需要你手动做

### 我可以直接在仓库里处理的内容

- 修改 workflow
- 修改 Worker 代码
- 修改部署文档
- 检查密钥名称是否与代码一致
- 在你完成控制台配置后继续排查报错

### 必须你自己手动完成的内容

- 在 GitHub 页面里添加 Secrets
- 在 Cloudflare 控制台里创建 R2 bucket
- 在 Cloudflare 控制台里创建 API Token
- 在 Cloudflare 控制台里给 Worker 绑定自定义域名
- 输入任何真实密钥、令牌、账号 ID

原因很简单：

- 这些操作涉及你的账号权限
- 这些步骤包含真实密钥
- 真实密钥不能进入仓库，也不能由我代填

## 1. 本项目当前使用的模型分工

当前仓库已经按阶段拆分了模型，不是一个模型跑完整条链路。

请按下面的值填写 GitHub Secrets：

```text
OPENAI_BRIEF_MODEL = gpt-5.4-xhigh
OPENAI_SCREENING_MODEL = gpt-5.4
OPENAI_ROSTER_MODEL = gpt-5.4
```

对应关系：

- `OPENAI_BRIEF_MODEL`：最终生成 `final.md`
- `OPENAI_SCREENING_MODEL`：中间筛选候选内容
- `OPENAI_ROSTER_MODEL`：账号打分

## 2. 第一步：把代码推到你的仓库

你的仓库已经创建好了：

```text
https://github.com/qinbing8/x-monitor
```

现在要确认两件事：

1. 本地仓库已经把 `origin` 指向这个地址
2. 你已经把当前代码推到 GitHub

如果 `.github/workflows/daily-report.yml` 还没有推上去，GitHub Actions 页面里不会出现 `daily-report`。

## 3. 第二步：设置 GitHub Secrets

### 去哪里设置

进入你的仓库后，按这个路径操作：

1. `Settings`
2. `Secrets and variables`
3. `Actions`
4. 进入 `Repository secrets`
5. 点击 `New repository secret`

注意：

- 当前方案使用的是 `Repository secrets`
- 不要求你创建 `production` environment
- 每一个 Secret 都要单独创建一次
- `Name` 必须和下面完全一致
- 保存后再次编辑时看不到原值，属于正常安全行为

### 需要创建的 Secret 名称

#### Grok

```text
GROK_API_KEY
GROK_BASE_URL
GROK_MODEL
```

#### GPT / OpenAI 兼容接口

```text
OPENAI_API_KEY
OPENAI_BASE_URL
OPENAI_BRIEF_MODEL
OPENAI_SCREENING_MODEL
OPENAI_ROSTER_MODEL
```

#### Cloudflare R2

```text
R2_ACCOUNT_ID
R2_ACCESS_KEY_ID
R2_SECRET_ACCESS_KEY
R2_BUCKET_NAME
```

#### Cloudflare Worker 部署

```text
CLOUDFLARE_API_TOKEN
CLOUDFLARE_ACCOUNT_ID
```

### 填写规则

- `*_API_KEY`：填真实 API Key
- `*_BASE_URL`：填对应服务接口地址
- `*_MODEL`：填模型 ID
- `R2_BUCKET_NAME`：填你的 bucket 名称
- `CLOUDFLARE_API_TOKEN`：填 Cloudflare API Token，不要填 Global API Key

## 4. 第三步：创建 Cloudflare R2 bucket

进入 Cloudflare 控制台：

1. 打开 `R2`
2. 点击 `Create bucket`
3. bucket 名称自己定，例如 `x-monitor-reports`
4. 创建完成

说明：

- bucket 名称不要求固定写死
- 当前 workflow 会在部署时把 `R2_BUCKET_NAME` 自动写入 Worker 配置
- 所以你不需要手工改 `worker/wrangler.toml` 里的 bucket 名称

## 5. 第四步：上传初始状态文件到 R2

进入刚创建的 bucket，上传下面这些对象。

### 必传

```text
state/X列表关注者.csv
```

### 可选

```text
state/account-score.json
state/X列表关注者.daily.csv
```

上传映射规则：

- 本地 `X列表关注者.csv` 上传到 `state/X列表关注者.csv`
- 本地 `account-score.json` 如果已有，就上传到 `state/account-score.json`
- 本地 `X列表关注者.daily.csv` 如果已有，就上传到 `state/X列表关注者.daily.csv`

说明：

- `account-score.json` 没有也能首次运行，系统会自己生成
- `X列表关注者.daily.csv` 没有也能首次运行
- `X列表关注者.csv` 是必需的主名单文件

## 6. 第五步：创建 Cloudflare API Token

你至少需要一组可以让 GitHub Actions 完成 Worker 部署的 Token。

进入 Cloudflare 控制台：

1. 打开 `My Profile`
2. 打开 `API Tokens`
3. 点击 `Create Token`
4. 创建一组用于 Worker 部署的 Token
5. 将值填入 GitHub Secrets：

```text
CLOUDFLARE_API_TOKEN
CLOUDFLARE_ACCOUNT_ID
```

注意：

- Token 只放 GitHub Secrets
- 不要写进 `wrangler.toml`
- 不要写进 Markdown

## 7. 第六步：手工触发第一次运行

不要先等定时任务，先手工跑一次。

进入仓库：

1. 点击 `Actions`
2. 找到 `daily-report`
3. 点击 `Run workflow`
4. 选择主分支
5. 点击确认运行

如果你看不到 `daily-report`：

- 先确认代码已经推到 GitHub
- 重点确认 `.github/workflows/daily-report.yml` 已经在默认分支上

## 8. 第七步：首次运行成功后检查 R2

如果 workflow 成功，R2 里最终应该能看到这些文件：

```text
reports/latest.json
reports/index.json
reports/YYYY-MM-DD/run-<run-id>/final.md
reports/YYYY-MM-DD/run-<run-id>/final.html
reports/YYYY-MM-DD/run-<run-id>/analyze.result.json
reports/YYYY-MM-DD/run-<run-id>/fetch.result.json
```

同时 `state/` 下的文件也会被回写更新：

```text
state/X列表关注者.csv
state/account-score.json
state/X列表关注者.daily.csv
```

## 9. 第八步：部署并绑定 Cloudflare Worker 域名

当前仓库已经有 Worker 代码和部署步骤。

注意：

- 只有你手工点击 `Run workflow` 的那次运行，才会执行 Worker 部署
- 每天定时跑的 `schedule` 任务只更新 R2 数据，不会重复部署 Worker

所以第一次上线时，请一定手工运行一次 workflow；成功后再做域名绑定：

1. 打开 Cloudflare 的 `Workers & Pages`
2. 进入部署出来的 Worker
3. 打开 `Settings`
4. 找到 `Domains & Routes`
5. 点击 `Add Custom Domain`
6. 绑定你的子域名，例如 `report.example.com`

绑定成功后，你会得到这几个访问入口：

- `/`：最新日报 HTML 页面
- `/history`：历史日报列表
- `/raw/latest`：最新 Markdown
- `/reports/YYYY-MM-DD/run-<run-id>`：指定某次 HTML 日报
- `/raw/YYYY-MM-DD/run-<run-id>`：指定某次 Markdown 日报

## 10. 当前 workflow 实际在做什么

现在仓库里的 `.github/workflows/daily-report.yml` 会按这个顺序执行：

1. 拉取仓库代码
2. 准备 Node.js
3. 生成临时凭据文件
4. 从 R2 下载已有状态文件
5. 运行 `node scripts/run.mjs --mode run`
6. 运行 `node scripts/publish-report.mjs`
7. 把 `final.md`、`final.html`、`latest.json`、`index.json` 上传回 R2
8. 如果这次是你手工点击的 `Run workflow`，再额外执行 `wrangler deploy` 部署 Worker

也就是说：

- 电脑关机不影响执行
- 本地不需要常驻开机
- 真正定时执行的是 GitHub Actions，不是你的电脑

## 11. 出错时先看哪里

### 场景一：GitHub Actions 一开始就失败

先查：

1. Secret 名称是否写错
2. 仓库默认分支上是否真的有 `daily-report.yml`
3. `R2_ACCESS_KEY_ID` / `R2_SECRET_ACCESS_KEY` 是否可用

### 场景二：workflow 成功，但 R2 没有日报

先查：

1. `scripts/run.mjs` 这一步是否真的产出了 `run-summary.json`
2. `scripts/publish-report.mjs` 是否执行成功
3. R2 bucket 名称是否和 `R2_BUCKET_NAME` 一致

### 场景三：域名可以打开，但首页没有内容

先查：

1. `reports/latest.json` 是否存在
2. `latest.json` 里的 `reportKey` 是否指向真实存在的 `final.html`
3. Worker 是否已经绑定到正确的 R2 bucket

## 12. 安全规则

以下内容不要提交到仓库：

- 真实 API Key
- `.env`
- Cloudflare Token
- R2 Access Key / Secret Key
- 任何包含完整密钥截图的文件

提交前至少检查这些文件里没有真实值：

```text
.github/workflows/daily-report.yml
worker/wrangler.toml
config.example.json
```

正确做法应该是：

- 仓库里只有变量名
- 真实值只放在 GitHub Secrets 和 Cloudflare 控制台

## 13. 验收标准

第一次跑通后，至少满足下面这些条件：

1. `Actions` 页面中 `daily-report` 是绿色成功
2. R2 中已经出现 `reports/latest.json`
3. R2 中已经出现当日 `final.md` 和 `final.html`
4. 自定义域名首页可以打开最新日报
5. `/history` 页面可以打开历史日报列表

满足这 5 条，就表示这套“电脑关机也能自动看日报”的链路已经跑通。
